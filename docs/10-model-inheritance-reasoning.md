# 业务模型继承 vs 从零建模比例推演

> **文档版本：** v1.0  
> **推演日期：** 2026-07-11  
> **推演背景：** DDF 平台支持重新建模和业务模型继承改编两种方式。当业务专家已在平台上构建了覆盖行业制造执行管理的标准业务模型后，用户会如何选择？  
> **推演方法：** 从产品设计特征、用户群体动机、行为经济学三个维度交叉推导

---

## 一、产品设计对「继承」的刻意倾斜

DDF 的三层模型结构本身就是为继承设计的：

```
base/           →   industries/      →   projects/
(通用实体框架)      (行业Profile)        (项目实例)
    继承链头           继承链中             继承链尾
```

叠加的产品机制形成正向推力：

| 产品特性 | 对继承的影响 | 对从零建模的影响 |
|----------|:-----------:|:---------------:|
| 三层模型继承（base→industry→project） | 强推。每个 Profile 声明 `extends` 基类 | 不禁止，但无继承好处 |
| NLP→DSL 自然语言建模 | 既降低继承后修改门槛，也降低从零建模门槛 | 双向降低，但**继承后改比从零写更快** |
| 模型市场分成（50%-70%） | 继承后贡献新 Profile 有经济激励 | 从零建模贡献新 Profile 激励更高（100% 原创者身份） |
| 交付即分叉（custom/ 不回灌） | 消除「继承后改不动」的恐惧 | 没有加分 |
| 模型市场的「精选认定」机制 | 高质量模型被继承 → 反向激励更多贡献 | 从零建模需要 ≥3 项目验证才能精选 |

**结论：产品设计在推「继承 > 从零」，但不是强制。继承是默认路径，从零是增值路径。**

## 二、按用户群体拆解动机

### 2.1 中小企业（目标占比 ~50%）

| 维度 | 倾向 |
|------|------|
| 建模能力 | 无。懂业务但不会 DSL，也不打算学 |
| 需求特征 | 「买个能用的，改改就能跑」 |
| 时间压力 | 大——月度订阅，越快上线越早产生价值 |
| 预算约束 | 紧——买标准版，不会额外花钱定制 |
| 大数定律 | 100 个中小企业里，流程相似度 80%+ |

**判断：95% 继承 / 5% 从零**

中小企业不会建模。标准 Profile 覆盖 80% 需求他们就满意了，改改字段名、配一下权限就上线。从零开始意味着「自己再发明一遍 MES」——不是他们的选项。

那 5% 的从零是：非常特殊的细分行业（如精密医疗器械），标准 Profile 覆盖率 < 50%，不得不从零。

---

### 2.2 行业集成商（目标占比 ~25%）

需分两个角色：

**角色 A：集成商作为交付方（项目交付场景）**

| 维度 | 倾向 |
|------|------|
| 核心诉求 | 快速交付，赚差价 |
| 对已有 Profile 的态度 | 行业 Profile 越完整，交付越快 |
| 角色转换 | 早期纯消费者 → 后期变成生产者（建新 Profile 卖） |

**判断：95% 继承 / 5% 从零**（交付已有 Profile 的行业客户时）

**角色 B：集成商作为 Profile 贡献者（模型市场生产者）**

| 维度 | 倾向 |
|------|------|
| 核心诉求 | 建一个 Profile → 卖 N 次 |
| 对 base/ 的态度 | 继承 base/ 层的通用实体框架（实体、关系、状态机基类） |
| 原创占比 | 真正的业务逻辑 100% 自己写，但继承 base/ 的通用骨架 |

**判断：20% 继承 / 80% 从零**（但「从零」是相对于行业 Profile，base/ 层仍继承）

**关键转折点：**
集成商一开始是纯消费者（继承现有的 Profile），等看到模型市场的分成回报后，会主动变成生产者（建新 Profile）。这个角色转换是 DDF 从「工具」变成「平台」的临界点。

**集成商整体加权：70% 继承 / 30% 从零**（早期继承多，后期建新 Profile 多）

---

### 2.3 大企部门级（目标占比 ~15%）

| 维度 | 倾向 |
|------|------|
| 需求定制度 | 高——要对接集团 ERP、OA、HR、MES |
| 技术能力 | 有内部 IT 团队，能写 YAML |
| 时间压力 | 中——优先保证与集团系统兼容 |
| 对行业 Profile 的态度 | 作为起点，但会大量修改流程层 |
| 定制深度的上限 | 对接层代码（custom/）可能比 generated/ 代码还多 |

**判断：75% 继承 / 25% 从零**

他们大概率继承行业 Profile 作为底座，但在以下方面有大量定制：
- 对接集团系统的 custom/ 集成代码
- 不同子公司的权限体系差异
- 集团特有的审批流、报表格式

那个 25%「从零」不是真从零——是从 `base/` 开始，不套行业 Profile。因为行业 Profile 里预置的流程与他们公司的实际流程不匹配，继承后改的工作量 ≈ 从零建一个。

---

### 2.4 AI Agent 开发者（目标占比 ~10%）

| 维度 | 倾向 |
|------|------|
| 核心诉求 | 一个「Agent 能操作的业务底座」 |
| 对 DSL 的态度 | 不是终点，是中间层。Agent 通过 MCP 调模型，不关心底层 YAML |
| 关注点 | 模型的完整性（是否覆盖所有 CRUD 操作 + 状态机） |
| 时间分配 | 80% 写 Agent 逻辑，20% 调模型 |

**判断：80% 继承 / 20% 从零**

他们要的是「现成的结构化业务上下文」。标准模型能提供这个，直接继承，然后通过 MCP API 写 Agent 逻辑。从零建模对他们的价值很低——浪费时间，不如直接继承开干。

那 20% 从零是：Agent 需要某种全新的、标准模型没有覆盖的操作语义（如某种 IoT 设备特有的状态机）。

## 三、加权汇总

| 用户群 | 预估占比 | 继承 | 从零 |
|--------|:-------:|:----:|:----:|
| 中小企业 | 50% | 95% | 5% |
| 集成商 | 25% | 70% | 30% |
| 大企部门 | 15% | 75% | 25% |
| Agent 开发者 | 10% | 80% | 20% |
| **加权合计** | **100%** | **≈ 85%** | **≈ 15%** |

**计算过程：**
- 继承 = 0.50 × 95% + 0.25 × 70% + 0.15 × 75% + 0.10 × 80%
- 继承 = 47.5% + 17.5% + 11.25% + 8% = **84.25%**
- 从零 = 100% - 84.25% = **15.75%**

## 四、这个比例随时间的变化曲线

| 时间轴 | 继承 : 从零 | 核心驱动力 |
|--------|:----------:|-----------|
| **Phase 0-1（模型市场未成熟，< 20 个模型）** | 70:30 | 没有现成 Profile 可继承，集成商必须自己建。从零比例最高 |
| **Phase 1-2（50+ 社区模型，精选机制上线）** | 85:15 | 多数常见场景已有 Profile，直接从市场拿。从零快速下降 |
| **Phase 2+（500+ 模型，精选认证可信）** | 90:10 | 模型质量经过认证，继承是默认动作。从零只剩「新行业 + 专家级深度定制」 |
| **稳定期** | **88:12** | 12% 的从零是「新行业 + 专家级深度定制」，这是平台内容增量的引擎 |

**关键洞察：** 从零比例不是单调递减的。每当平台进入一个新行业，从零比例会短暂反弹（因为该行业没有 Profile），然后随模型积累再次下降。

## 五、核心结论

### 5.1 数字结论

**继承 ≈ 85% vs 从零 ≈ 15%** 不是拍脑袋——是以下因素共同作用的结果：

1. 产品设计（三层继承 + 模型市场分成 + 交付即分叉）天然推继承
2. 目标群体结构（中小企业 + 集成商占 75%）天然偏好继承
3. 从零的门槛（DSL 语法 + 行业知识）天然筛选用户

### 5.2 那 15% 从零的价值

那 15% 的从零不是浪费——**正是这 15% 在持续产生新行业 Profile，养活剩下 85% 的继承用户。**

- 每个从零建模的集成商都在为平台增加一个可继承的 Profile
- 每个精选 Profiles 形成后，被继承的频次会指数增长
- 从零 = 平台的「增量引擎」，继承 = 平台的「复制引擎」

**模型市场的飞轮闭环就靠这 15% 的贡献者在转。**

### 5.3 产品策略建议

| 针对群体 | 策略 |
|---------|------|
| 中小企业（95% 继承） | 默认推「推荐 Profile」→ 一键继承 → 拖拉拽改字段 → 上线。不要让他们看到 DSL 编辑器 |
| 集成商（70% 继承 → 30% 从零） | 展示「模型市场分成收入」数据吸引他们从消费者变成生产者。降低从零建模的入门门槛（NLP→DSL + 模板克隆） |
| 大企部门（75% 继承） | 强化 custom/ 目录的对接能力，减少他们「从零建 Profile」的冲动 |
| Agent 开发者（80% 继承） | 强化 MCP 协议的文档和示例，让继承模型后 5 分钟出 Agent 逻辑 |
